What is the difference between Session and Cookie

Posted by HanTing on Thursday, January 23, 2020

Rails面試中其中之一最容易被問的問題，就是Session與Cookie的差別。Session和Cookie被官方文件，以及Ihower的實戰聖經歸類在Controller的文章中。

講到session和cookie，就要先從在網頁中，HTTP stateless的特性開始講述

所謂stateless，就是server無法在不同的request/response cycle中記住資訊。這篇以沒有記憶力的小明比喻HTTP stateless的特性。request對於server來說，每次都是全新的request一樣，server無法得知上個request的狀態。session和cookie在這邊扮演輔助記憶的角色。

Cookie 是把資料存在瀏覽器裡面，最常見到的 Cookie 應用是在表單填寫，假設在頁面上的表單資料填到一半，不小心把網頁關掉，這時再重新打開發現先前填的內容還在的話，靠的就是 cookie，這種沒有安全性的設定可以用字串被存在Cookie裡面。

在Rails中，有四種機制來儲存session

  1. ActionDispatch::Session::CookieStore ─ 所有資料都存在用戶端。
  2. ActionDispatch::Session::CacheStore ─ 資料存在 Rails 的 Cache。
  3. ActionDispatch::Session::ActiveRecordStore ─ 資料使用 Active Record 存在資料庫（需要 activerecord-session_store RubyGem）。
  4. ActionDispatch::Session::MemCacheStore ─ 資料存在 memcached（這是遠古時代的實作方式，考慮改用 CacheStore 吧）

在Rails中預設使用 CookieStore 儲存session。這個選項最大的不同在於會將所有的session資訊經過加密後，存在cookie當中，Server可以直接透過cookie得到session的資料，而不是透過存取session ID 到伺服器上查詢 Session 資料。存在 Cookie 的資料經過加密簽署，防止有心人士竄改。即便是擁有 Session 資料存取權的人，也無法讀取內容（內容經過加密）。如果 Cookie 的資料遭到修改，Rails 也不會使用這個資料。官方文件1、官方文件2有更詳細的解釋。

CookieStore 大約可以存 4KB 的資料，其他儲存機制可以存更多，但通常 4KB 已經夠用了。不管用的是那種儲存機制，不建議在 Session 裡存大量資料。特別要避免將複雜的物件儲存在 Session 裡（除了 Ruby 基本物件之外的東西都不要存，比如 Model 的實體）。因為伺服器可能沒辦法在請求之間重新將物件還原，便會導致錯誤發生。

若使用者的 Session 沒有儲存重要的資料，或存的是短期的資料（比如只是用來顯示提示訊息）。可以考慮使用 ActionDispatch::Session::CacheStore。這會將 Session 存在應用程式所設定的快取裡。優點是利用現有的快取架構來儲存，不用額外管理，或是設定 Session 的儲存機制。缺點是生命週期短、隨時可能會消失。

選用哪一個通常可以用安全性來判斷，需要安全性的就用Session，沒安全顧慮的就用Cookie。
